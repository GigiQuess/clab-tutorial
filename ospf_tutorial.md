OSPFの最大のメリットは、「ネットワーク障害時に自動で迂回ルートを探し、通信を維持する（自己修復）」ことです。
これを実感するために、先ほどの3台構成を「直線」から「三角形（冗長構成）」に変更し、1箇所をわざと切断しても通信が続く様子を体験するチュートリアルを作成しましょう。
------------------------------
1. トポロジーの変更（三角形にする）
R1とR3を直接つなぐリンクを追加します。
ospf-lab.clab.yml のリンク部分を修正:

  links:
    - endpoints: ["r1:eth1", "r2:eth1"] # ルートA
    - endpoints: ["r2:eth2", "r3:eth1"] # ルートA
    - endpoints: ["r1:eth2", "r3:eth2"] # 追加：ルートB（ショートカット）

2. 各ルータの frr.conf 更新
新しく追加したリンク（eth2）にもIPとOSPFの設定を入れます。

* R1: eth2 に 10.0.13.1/24 を追加、network 10.0.13.0/24 area 0 を追加。
* R3: eth2 に 10.0.13.3/24 を追加、network 10.0.13.0/24 area 0 を追加。
* R2: 変更なし。

------------------------------
3. メリット実感チュートリアル・ステップステップ1：正常時の経路を確認
まず、R1からR3（10.0.23.3）への経路を確認します。

# R1で実行
docker exec -it clab-ospf-frr-lab-r1 vtysh -c "show ip route"

注目点: 10.0.23.3 への経路が、R2経由ではなく、直接つながった R3(10.0.13.3) 経由になっているはずです（OSPFは「コスト」が低い最短ルートを自動で選びます）。

ステップ2：継続的な通信を開始
R1からR3へpingを打ち続け、通信が生きていることを監視します。

docker exec -it clab-ospf-frr-lab-r1 ping 10.0.23.3

ステップ3：【障害発生】メインルートを切断
別のターミナルを開き、R1とR3を直接結んでいるインターフェースを無効化（リンクダウン）させます。

docker exec -it clab-ospf-frr-lab-r1 ip link set eth2 down

ステップ4：自動切り替えの観測
pingを打っている画面を見てください。

   1. 一瞬だけ Destination Host Unreachable などが出るか、止まります。
   2. 数秒後、再びpingが通り始めます。

ステップ5：迂回ルートの確認
再び経路を確認します。

docker exec -it clab-ospf-frr-lab-r1 vtysh -c "show ip route"

注目点: 今度は 10.0.23.3 への経路が、R2（10.0.12.2）経由に自動で書き換わっています！

------------------------------
このチュートリアルで学べるメリット

* 動的な経路制御: 人間が設定を変えなくても、ルータ同士が「道が壊れた！」と情報を交換し、勝手に代わりの道を探してくれます（スタティックルートではこうはいきません）。
* 最短経路の選択: 常に最適なルートを計算して通信効率を高めてくれます。
